Ubuntu typography project recommendations

Neale Willis

Improving typography in Ubuntu involves many things, not just choices of font. The effect of typography is dependent on use of color, horizontal spacing, line spacing, and layout of windows as a whole. So this project has touched on some of those issues.

This document has two parts. First, there are recommendations to Ubuntu developers. These are things we should change in the font engine default settings, the Ubuntu theme, the toolkits, and default software such as Unity and Nautilus.

Second there are guidelines for application developers, which will be published separately when finished. We may need to do some IDE and/or toolkit work to make it easy for application developers to follow these guidelines.

Recommendations to Ubuntu developers

Most of these recommendations involve changes in the default theme. Some involve changes to default font preferences, font engine settings, and toolkit widgets, and individual elements in Unity and other system components.

Base font size

Ubuntu 12.04 uses a larger base size than our biggest competitors.

Windows XP, 7, 8

11 px

Mac OS X 10.7

13 px

Google Chrome

12 px

Ubuntu 12.04

15 px

This sizing causes interface elements to take more space, leaving less space for information.

We should ☐ reduce the base font size by 1px to 14px[a], to reduce visual clutter and increase screen real-estate. Ubuntu is a robust font which benefits from wide leading – displayed at large sizes with tight leading leads to degraded legibility. A slightly smaller font size with more leading optimises legibility in this instance.

How do we derive this scale?

Ubuntu 12.04 also utilises both the most font variations and oddly, the least in a given area of the comparison systems. Nine variations in font size and weight can be identified within the Dash with up to six being in use at any one time. In contrast, a single font size is used throughout the desktop. Comparing this to the variations observed in the competitor analysis, deliberate and restrained use of font size can aid in creating greater visual rhythm, pattern and harmony.

On occasion, sizing falls outside these recommendations, hindering rhythm, hierarchy and visual harmony.

We can bring a visual harmony to the text used in an interface through the use of a ratio based system to create a typographic scale appropriate to the given project.

An online tool created by Tim Brown, Type Manager for Adobe Typekit, modularscale.com, helps us to create a modular scale based on a key number relevant to the given layout or project and a series of established ratios.

Creating a modular scale using this system to give us an harmonious appearance when using 14px as the base font size and 12px, a suitable size for small text, as our important number with a ratio of 1:√2 – Aug. fourth/dim. fifth or 1:1.414, creates the following scale when rounded to the nearest whole number:

7, 8, 10, 12, 14, 17, 23...

Screen grab of the proposed scale as taken from ModularScale.com:

By introducing an extra measure at the halfway point between Body and Small, thereby creating additional flexibility, we arrive at the following scale:

17px         Headline

14px         Body

13px         List

12px         Small

10px         Extra small

The above uses 14px as the base font size, from which the remaining sizes are calculated. It could therefore be stated, in this instance, 14px is equal to 1em. If the remaining sizes are then calculated as ems, the following is true:

1.212         Headline

1                Body

0.928        List

0.857        Small

0.707        Extra small

Using the above calculation, a consistent rhythm to font sizing can be employed regardless of the intended application. For example, design for TV requires much larger font sizes than that for desktop. Employing the above scaling will allow for the creation of a typographic scale that is consistent to both TV and desktop regardless of the starting base size as the headline is always 1.212 em or 1.212 times larger than the expressed base.

Finally, typographic sizes should be measured using ems, rather than specified pixels to allow greater flexibility when applying scale to other devices.

Font hinting

Ubuntu’s base size of 15px is currently being resized by the system’s auto-font hinting, set by default to Medium, to 14.6px. This reduces the size of capital letters by 2 pixels and brings about a reduction in the width of elements – a drop down menu can be seen to change from 328px in width with auto-hinting set at medium, to 340 with auto-hinting removed.

We should ☐ turn off auto-hinting, at least for the Ubuntu font, thereby returning the sizing and spacing of elements to their intended values.

Line spacing

The positioning of text within a given area can affect legibility and how the text is read. Similarly, the positioning of elements in relation to text can affect how they are interpreted. By basing the size of this spacing around the em size, as defined previously, a visual harmony and rhythm can be attained for the entire interface, both in relation to vertical and horizontal positioning.

Vertical positioning

A vertical rhythm – the spacing and arrangement of text as the reader descends the page – guides and engages the reader as they move through a page or interface. A consistency applied to an elements vertical spacing, based around a common line height vastly improves legibility.

The Ubuntu font requires fairly wide line spacing to achieve the most legibility. Line spacing of 1.3em for body text creates a visually pleasing layout. Therefore, applying this thinking to a base size of 14px equalling 1em, the above would give a baseline of 18px.

Theme

Colors

Examples of light grey text used on an orange background can be found throughout the interface. This colour combination has insufficient contrast to be considered readable or accessible.

Use of black, or dark grey, text on an orange background is not ideal for legibility. It is close to black on red, which strains the eye and is far from ideal for the colour-blind.

Inactive states of key code options within drop down menus is denoted by greyed out text, presently set in a very low contrast tone when compared to the background grey. This combination of grey tones fall outside recommended contrast levels for legibility and are therefore hard to read, sometimes invisible on poorly calibrated monitors.

Where row has an orange background, white text should be employed.

General use of colour and contrast between used colours needs closer attention, with specific thought toward correcting dark grey on orange.

Menus

Presently, menu bar menus have large, 8px spacing between menu items which can result in surprisingly long menus. Dividers used within the menus are also incorrectly positioned at 1px above the central point between the menu items they separate, causing a curious unbalanced visual effect.

Recommendation

As an exception to the recommended typographic scale, these menus should employ a slightly larger font of 15px. The use of a larger font brings the menu contents to fore, differentiating it from background clutter such as wallpaper, multiple open windows and applications etc.

Other typographic principles should remain the same, such as leading of 1.3em of the base size used for the element (therefore 15px font size == 20px leading) between menu items. Where a divider is appropriate, leading of 30px should be used. Applying this leading could reduce menu height from 462px to 423px while improving legibility. Padding around elements should also follow previously outlined sizings – left: 1em for icon, plus ½em padding, right: ½em.

Dividers should be aligned to the vertical centre between menu items.

Buttons[c]

Horizontal padding

The sizing of buttons within Ubuntu is relatively arbitrary, with differing amounts of padding between button edge and text. Clusters of buttons sized to equal that containing longest text can lead to layout issues should the smallest text differ greatly from the longest.

We should standardize button padding so that the space between the text and the button edges is 1 em, unless the length of text within the button exceeds a given size whereby the padding decreases to ½em[d].

Automatic button width

Buttons should be of equal size[e] unless the length of the longest far exceeds[f] that of the shortest. The button size should then be determined by the next longest text unless the above is true, until the sizing reaches the shortest text plus 1em.

This method of calculating spacing can also be applied to padding around text within elements and placement of elements to text.

Spacing within buttons should be 1em from the edge of the button the text, around text in tabs and the space from icon to text in a list or menu should be ½em.

Dividers

Follow baseline grid. Dividers occupy a space which fits within the suggested baseline. Dividers can take the form of a blank spacer or line which stretches across the affected area.

Alert boxes

Various messages are communicated through the use of a pop-up dialog box, the messages within containing a varying degree of severity, from save confirmations to internal errors within Ubuntu.

The primary, or alert, message is in direct competition with the boldness of the dialog title and orange primary button action, creating a confused taxonomy.

A seemingly arbitrary approach to horizontal sizing leads to wildly differing sizes of dialog box. In turn, this leads to both very short and very long lines of copy, each with their own effect to readability. Referring to the suitable maximum line length of 33em, we can begin to make considerations of an appropriate horizontal sizing for dialog boxes so that text contained within is of a comfortably readable length.

Applying the proposed baseline structure to the layout helps create visual harmony and make the message easier for the eye to break down.

By then addressing the taxonomy of the dialog box by applying the proposed typographic scale and use of font weight, we see the primary message as the only text set in Bold and at a larger size. Therefore, this clearly becomes the dominant element.

For instances where there is no alert message[g], the reduced boldness of the dialog title aids the primary button selection to become the clearly dominant element.

Nautilus improvements

Presently, the same size font is used throughout the File Management Window – the title bar is the same size as the file names, the breadcrumb the same as the navigation. This creates a consistent, easily deployed and managed approach to the window but does make the window harder to visually digest – the eye skips around looking for something to get hold of, leading to the question of hierarchy and ‘just what is important in this element?’.

Taking a moment to revisit the competitor analysis to look at how Google Chrome and Apple OS X address similar windows, we can see they have some success in creating a visually digestible window. Closer inspection reveals they employ slight font size variations across the window to bring a hierarchy to the window and allow the user to break down the window quickly.

With the present Ubuntu window, the entire file management window appears as a single unit with little differentiation between areas. 3 areas of bold text are evident – the window title, the current location in the breadcrumb and the headers of each section of navigation. Everything else appears equal in weight and importance making the sections of the window appear identical.

The file management window can be broken into distinct 4 areas of information – window title, navigation, breadcrumb and file management. Looking at the current window, the order of hierarchy applied to these information areas are, although slight, appear to be:

1. Window title

2. Navigation

3. Breadcrumb

4. File management

Thinking about the intended use of the file management window, it could could be assumed the taxonomy of the window would follow the order of management of files, followed by navigation, breadcrumb and window title.

1. File management

2. Navigation

3. Breadcrumb

4. Window title

Guidelines for application developers

Introduction

Typography reaches beyond the choice of font. Effective typography creates effective communication through the considered use of emphasis, vertical scale and sizing and spacing.

When using type, the designer should consider the user and what is important for them to understand and what is important for you to communicate at each given point, avoiding the tendency to shout ever louder. Applying a consistent use of font weight, sizing and padding creates an effective structure and taxonomy on which to communicate the desired message.

The below document details methods of building effective communications around a solid structure through the application of guidelines, including:

  1. The creation of an effective, yet flexible typographic scale and therefore effective emphasis through considered use of font weights and sizing.
  2. Introducing the use of Medium to replace many current instances of Bold use and removing the use of italics, thereby restricting the use of Bold to instances where emphasis is truly required.
  3. Proposal of a method creating a consistently visually harmonious structure to areas in which type exists, applying to vertical rhythm, line length, leading and padding within elements, based around the base font size.

Pixel        Em        Name

17px        1.212        Headline

14px         1        Body

13px        0.928        List

12px        0.857        Small

10px        0.707        Extra small

Typographic sizes should be measured using ems, rather than specified pixels to allow greater flexibility when applying scale to other devices.

Bold and other font weights

Used to give importance to a piece of text, Ubuntu Bold should be used sparingly and only where it is necessary. Examples throughout Ubuntu use bold then large bold to differentiate ideas and bring emphasis to elements. This rapidly leads to situations where bold is no longer strong enough emphasis and the use of italic or larger text is used to differentiate.

Ubuntu Bold is a very heavy, thick set font – a thickness can lead to reduced legibility at relatively small sizes. The heaviness of the font face also creates a darker, heavier tone to the interface.

The historically intended use of bold is to strongly emphasise a particular piece of information and so therefore, Bold should only be applied where a piece of information requires significant emphasis. Careful thought of what is being communicated at the given point and restrained use of Bold, restricted to specific areas such as error description in dialog boxes, can aid to create a dramatic message where and when it is needed.

Medium should be used in most cases where Bold is currently employed, for example list headers or window titles. This creates a significant differentiation of font style while reserving Bold for important messages.

Italics

Historically used to bring attention to a word or piece of text, italics are used for reasons such as straight emphasis, introducing or defining terms or titles of standalone works. Within Ubuntu, italics are arbitrarily applied, most often used as a method of devaluing a given piece of text – a selected or default item in a list or a piece of help text, for example.

Recommendation

Italics shouldn’t be used within the system panels, menus or interface. Italic text is generally harder to read and not significantly different to regular text to be used as an emphasis within the interface.

An argument for using italic to set text field hint copy can be made.

Priority and hierarchy

What is important?

A key question to ask when constructing any view, panel or element is, ‘just what is important at this point?’ or to put it another way, ‘what do I, as designer of this element, want the user to be looking at at this moment?’.

Throughout the Ubuntu desktop, the generally uniform font size creates a consistent appearance but in cases where something needs to be communicated to the user, how can the designer or developer go about making something stand out? At present there are numerous examples of text being made heavier, larger, then large and bold or perhaps italic in an attempt to address this. Following along this path, where does it end?

Of course, this does not apply to alerts but to each element. By asking the question of ‘what is important here’, correctly applying a hierarchy to the information contained within any element will improve the ability of the user to understand what is being shown or communicated to them at any given point – be it a drop down menu to a file management window to a system critical alert dialog.

Differences can be subtle but when used in conjunction make significant impressions on the legibility of an element.

Horizontal positioning

Possible arbitrary placement of elements, particularly evident within system panels, can greatly affect the ability of the user to understand the message that’s being attempted to be communicated. Similarly, the length of a line of text can affect the ability to read a given message – should a line be too long, the reader can have difficulty reading the text.

Recommendation

This measure gives us an ideal opportunity to create a series of guideline sizes for elements containing text for use in areas such as system panels.
